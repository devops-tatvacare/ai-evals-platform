# DEVOPS-DETAILS — AI Evals Platform

Production deployment + operational runbook. Self-contained. Each section is structured as **what / why / how** so anyone on call can act without searching elsewhere.

For local setup and the full env var surface, see `docs/SETUP.md`.

---

## 1. Production topology

**What.** Two Azure Container Apps backed by an external Postgres and Blob Storage.

| Component | Resource |
| --- | --- |
| Backend | Container App `ai-evals-be-prod` (image from ACR) — runs FastAPI + worker loop in-process |
| Frontend | Container App `ai-evals-fe-prod` (image from ACR) — nginx + SPA |
| Database | Azure Database for PostgreSQL (external) |
| Files | Azure Blob Storage (external) |
| Images | Azure Container Registry `aiexperiment.azurecr.io` |
| Resource group | `product` |

**Why.** Single backend container keeps the deploy story simple while we are pre-scale; the worker runs inside the API container (`JOB_RUN_EMBEDDED_WORKER=true` default). Once job volume justifies a split, add a second Container App for the worker and flip the backend's flag to `false`.

**How to inspect.**

```bash
az containerapp show --name ai-evals-be-prod --resource-group product
az containerapp revision list --name ai-evals-be-prod --resource-group product
az containerapp logs show --name ai-evals-be-prod --resource-group product --type console --tail 200
```

`az containerapp show` returns plaintext for any env var stored as `value:` — NEVER print this output into Action logs. Treat the field as sensitive.

## 2. CI/CD

**What.** Two GitHub Actions workflows in `.github/workflows/`:

- `ai-evals-be-prod-AutoDeployTrigger-*.yml` — backend
- `ai-evals-fe-prod-AutoDeployTrigger-*.yml` — frontend

**Why.** Each workflow is generated by Azure's Container Apps "auto deploy" wizard, OIDC-bound to a Service Principal whose federated credential subject is `repo:<owner>/<repo>:ref:refs/heads/prod`. Triggering from any other branch or with `environment:` set fails with `AADSTS700213`.

**How a deploy runs.**

1. Push to the `prod` branch (or run `workflow_dispatch` from `prod`).
2. Workflow checks out the repo, authenticates to Azure via OIDC using `secrets.AIEVALSBEPROD_AZURE_*`.
3. `azure/container-apps-deploy-action` builds the image from `backend/` (backend) or root (frontend), pushes to ACR, and updates the target Container App revision.

**How to deploy.**

```bash
# 1. Merge changes to main, verify tests pass.
# 2. Fast-forward prod -> main and push:
git checkout prod
git merge --ff-only main
git push origin prod
# Workflow auto-triggers. Watch in GitHub Actions.
```

**Required GitHub secrets.**

| Secret | Purpose |
| --- | --- |
| `AIEVALSBEPROD_AZURE_CLIENT_ID` | OIDC client id |
| `AIEVALSBEPROD_AZURE_TENANT_ID` | OIDC tenant id |
| `AIEVALSBEPROD_AZURE_SUBSCRIPTION_ID` | Subscription scope |
| `AIEVALSBEPROD_REGISTRY_USERNAME` | ACR pull/push |
| `AIEVALSBEPROD_REGISTRY_PASSWORD` | ACR pull/push |

(Frontend mirrors with `AIEVALSFEPROD_*`.)

**Rollback.** Container Apps keeps revision history. Roll back without a rebuild:

```bash
az containerapp revision list --name ai-evals-be-prod --resource-group product -o table
az containerapp revision activate --revision <previous-revision> --resource-group product
```

## 4. Runtime configuration

**What.** Container App env vars and secrets. See `docs/SETUP.md` for the full surface; this section calls out production-critical ones and the secret/value distinction.

**Why secret vs value matters.** Any var stored as `value:` is plaintext in `az containerapp show`, in revision diffs, and in any Actions log that dumps the spec. Anything sensitive MUST be `secretRef:`. Past leaks confirmed: `JWT_SECRET`, `ADMIN_PASSWORD`, `LSQ_*`, `AZURE_STORAGE_CONNECTION_STRING`, `GEMINI_SERVICE_ACCOUNT_JSON`, DB password — all are required to be `secretRef:`.

**How to set a secret correctly.**

```bash
# Create or update the secret on the Container App.
az containerapp secret set \
  --name ai-evals-be-prod --resource-group product \
  --secrets jwt-secret=<value>

# Bind the env var to the secret.
az containerapp update \
  --name ai-evals-be-prod --resource-group product \
  --set-env-vars JWT_SECRET=secretref:jwt-secret
```

**Required at first deploy.**

| Variable | secretRef? | Notes |
| --- | --- | --- |
| `DATABASE_URL` | yes | DSN incl. password |
| `JWT_SECRET` | yes | 64-char hex |
| `LLM_CREDENTIAL_KEY` | yes | Fernet key. MUST be set before `alembic upgrade head` runs — migration 0047 backfills encrypted credential rows. Loss = unreadable tenant keys. |
| `ORCHESTRATION_CONNECTION_KEY` | yes | Fernet key. Loss = unreadable provider connections. |
| `AZURE_STORAGE_CONNECTION_STRING` | yes | Blob auth |
| `AZURE_STORAGE_CONTAINER` | no | `evals-files` |
| `FILE_STORAGE_TYPE` | no | `azure_blob` |
| `CORS_ORIGINS` | no | public origin |
| `APP_BASE_URL` | no | public origin; drives webhook URL composition |
| `ADMIN_EMAIL` / `ADMIN_PASSWORD` / `ADMIN_TENANT_NAME` | password as secret | Only used for first-run bootstrap |
| `GEMINI_SERVICE_ACCOUNT_JSON` | yes | base64; entrypoint decodes to `/app/service-account.json`. Optional — system-tenant Sherlock fallback only. |

`LSQ_*`, `WATI_*`, `BOLNA_*`, `SMS_*` env vars are read once on first boot to seed default `ProviderConnection` rows. After that, runtime reads come from the DB. New deployments configure these via the in-app Connections page; the env values are advisory and planned for removal.

**Updating env vars in place.**

```bash
az containerapp update \
  --name ai-evals-be-prod --resource-group product \
  --set-env-vars KEY=value
```

A successful update creates a new revision and shifts traffic. To roll back, see Section 2.

## 5. Schema migrations

**What.** Alembic is the only schema truth. Migrations live at `backend/alembic/versions/`. `alembic upgrade head` runs on every container boot via `backend/entrypoint.sh`.

**Why.** Schema drift between code and DB is the most expensive bug class in this system. Past prod outages traced to unqualified raw SQL referencing renamed tables (CrashLoopBackoff) and to env vars missing before a migration needed them.

**How to ship a migration safely.**

1. Author the migration locally: `PYTHONPATH=backend alembic -c backend/alembic.ini revision --autogenerate -m "<message>"`.
2. Edit the generated file. Schema-qualify every raw SQL string (`platform.evaluators`, `analytics.fact_evaluation`). No bare table names — the search path is `"$user", public` and bare names crash.
3. Run end-to-end on a fresh local DB: `docker compose down -v && docker compose up --build`.
4. Run against a prod-like snapshot if the change is non-trivial.
5. Verify the matching ORM model is in the same commit.
6. Push to `prod` — Alembic runs on container start.

**If boot crashes on migration.** Container App shows `ProcessExited exit code 3` in system events. Diagnose by:

- Reading console logs for the failing SQL (`az containerapp logs show ... --type console`).
- Confirming `LLM_CREDENTIAL_KEY` and `ORCHESTRATION_CONNECTION_KEY` are set (validator runs before Alembic).
- Replaying locally with a read-only prod DSN (`SET default_transaction_read_only = on`) to surface pre-write errors without risking the DB.

## 6. Worker model

**What.** The worker is a job loop that claims rows from `platform.background_jobs`, leases them, executes, retries on failure.

**Why we run it in-process today.** No separate worker Container App exists. Defaulting `JOB_RUN_EMBEDDED_WORKER=true` keeps the same image serving both HTTP and background work.

**How it behaves.**

- Queue classes: `interactive`, `standard`, `bulk`, `analytics`. Independent caps via `JOB_*_MAX_CONCURRENT`.
- Per-tenant / per-app / per-user concurrency caps prevent a noisy tenant from starving others.
- Leases + heartbeats prevent duplicate execution on restart.
- Stale recovery runs on lifespan start: any job past `JOB_STALE_TIMEOUT_MINUTES` without a heartbeat is re-claimed.
- Orphaned `EvaluationRun` reconciliation runs for the inside-sales path.

**If jobs are not progressing.**

1. `az containerapp logs show ... --type console --tail 500` and grep for `JobRunner`.
2. Check `platform.background_jobs` row state (claimed / running / failed) via the DB.
3. Confirm `JOB_RUN_EMBEDDED_WORKER` is unset or `true` on the backend (the worker lives inside this container today).
4. If a flood of `analytics`-class jobs is blocking smaller work, raise `JOB_ANALYTICS_MAX_CONCURRENT` or schedule a maintenance burst.

## 7. File handling

**What.** Files (audio, CSV, exports) live in Azure Blob Storage in production.

**How.** Backend reads `FILE_STORAGE_TYPE`. `azure_blob` routes through the `AZURE_STORAGE_CONNECTION_STRING` + `AZURE_STORAGE_CONTAINER` pair. `MAX_UPLOAD_SIZE_MB` (default 100) caps requests. nginx in front accepts up to the same limit.

**If file-backed workflows fail.**

1. Confirm `FILE_STORAGE_TYPE=azure_blob`.
2. Validate the connection string by listing blobs from `az storage blob list`.
3. Confirm the container exists.
4. Confirm size + MIME limits in `MAX_UPLOAD_SIZE_MB` and `ALLOWED_UPLOAD_MIMES`.

## 8. Public routing + webhooks

**What.** Frontend Container App serves SPA on `:443`, reverse-proxies `/api/*` to backend `:8721`. The orchestration runtime publishes webhook URLs at `{APP_BASE_URL}/api/orchestration/webhooks/{provider}/{token}`.

**Why webhook URLs hide tenant/app.** The per-`ProviderConnection` token resolves both at receive time. This keeps URLs short, copyable, and rotatable.

**How to add a new tenant's provider.**

1. Tenant admin opens Connections, creates a `ProviderConnection`, copies the webhook URL.
2. Pastes the URL into the provider dashboard (WATI, Bolna, etc.).
3. Verifies inbound by `Run Now` on the cloned workflow.

## 9. Sherlock + LLM credentials

**What.** Tenant LLM credentials are encrypted at rest in `platform.tenant_llm_providers` (Fernet via `LLM_CREDENTIAL_KEY`). Sherlock falls back to a system-tenant Gemini service account when a tenant has no Gemini/OpenAI/Azure key.

**How.** Production decodes `GEMINI_SERVICE_ACCOUNT_JSON` (base64) into `/app/service-account.json` via `backend/entrypoint.sh`, and sets `GEMINI_SERVICE_ACCOUNT_PATH=/app/service-account.json`. If a real tenant fails, the right fix is to add their LLM key in Admin → AI Settings, not to fix the SA fallback.

## 10. Debugging prod without portal access

**What.** Common failure mode: backend will not start, no Azure portal access in your account.

**How.** The deploy Service Principal can run read-only `az containerapp ...` commands via a one-off GitHub Action that reuses the existing OIDC federated credential.

1. Author a temporary workflow with `on: workflow_dispatch` that runs the read-only commands you need (`az containerapp show`, `revision list`, `revision show`, `logs show --type system`, `logs show --type console`). DO NOT include any step that dumps env vars.
2. Push the workflow file to the `prod` branch (federation subject is bound there). Trigger it with `gh workflow run <file> --ref prod`.
3. Output appears in the Actions log.
4. DELETE the workflow file once the diagnosis is done.

**Local lifespan replay.** Open a Postgres session against the prod DSN, `SET default_transaction_read_only = on`, then call each lifespan step (`alembic upgrade head` in plan mode, `sync_column_comments`, `seed_all_defaults`, etc.). Postgres rejects writes with `cannot execute INSERT/UPDATE/DELETE/COMMENT in a read-only transaction`. Pre-write failures (`UndefinedTableError`, missing column, `ImportError`, env validator) are real bugs you can diagnose without touching the container.

**Recognise these root causes.**

| Symptom | Likely cause |
| --- | --- |
| `UndefinedTableError: relation "X" does not exist` | Raw SQL referencing a renamed/unqualified table. Schema-qualify it. |
| `permission denied for relation X` on `COMMENT` or DDL | Connecting role is not owner of the target table. |
| `RuntimeError: <ENV> environment variable is required` | `validate_startup_config` failed. Set the missing var as `secretRef`. |
| `ImportError` deep in `app.services.job_worker` | A handler module imports a removed model. |
| `cannot drop column ... because other objects depend on it` | View or FK references the column. Migration needs `op.drop_constraint` first. |

## 11. Operational checklist (first deploy)

1. Provision Azure resources (Section 3).
2. Bind the OIDC federated credential to `repo:<owner>/<repo>:ref:refs/heads/prod`.
3. Create ACR pull/push secrets in GitHub.
4. Set required env vars + secretRefs on the backend Container App (Section 4).
5. Push to `prod` branch — CI deploys.
6. `curl https://<host>/api/health` returns `{"status":"ok","database":"connected"}`.
7. Log in as bootstrap admin, seed evaluators per app, create roles + app grants, generate invite links, rotate bootstrap password.
8. Tenant rollout via the in-app Connections page (see SETUP §7).
